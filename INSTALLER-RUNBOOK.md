# Agentic Config Kit Installer Runbook

Use this runbook when a user asks to set up the Agentic Config Kit in another
repository. It is written for an assisting model or coding agent to execute on the
user's behalf.

## Goal

Bring a target repository up to the standard Agentic Config Kit setup:

- canonical shared assets in `.ai/`;
- generated local projections for Cursor, Codex, Claude Code, Windsurf/Devin, and Continue;
- source-only Git hygiene for generated IDE folders;
- root `AGENTS.md` bootstrap guidance;
- built-in `agentic-config-maintainer` skill and `/agentic-config` command;
- verification with `doctor`, `--check`, and optional pre-commit guard.

## Inputs

Required:

- Target repository path.

Optional:

- Whether to install the Git pre-commit hook.
- Whether to replace starter example assets with repo-specific seed assets.
- Initial repo guidance, commands, skills, or agents the user wants captured.

If the target repository path is missing, ask for it. Otherwise prefer making safe
defaults and continue.

## Safety Rules

- Do not overwrite an existing `.ai/` directory. If one exists, stop and inspect it.
- Do not edit generated files with `AUTOGENERATED` markers by hand.
- Do not delete existing IDE-native config unless the user explicitly asks.
- Do not treat `.codex/rules/*.rules` as portable behavioral guidance; those files are
  Codex execution policy.
- Prefer source-only mode: commit `.ai/`, `AGENTS.md`, `.ai/.manifest.json`,
  `sync-agentic.sh`, and any project docs/hooks; keep generated IDE projections ignored.

## Step 1: Inspect The Target Repo

Run from this kit repo:

```bash
pwd
test -d /path/to/target-repo
test -d /path/to/target-repo/.git
find /path/to/target-repo -maxdepth 2 \( -name .ai -o -name sync-agentic.sh -o -name AGENTS.md -o -name .cursor -o -name .agents -o -name .claude -o -name .windsurf -o -name .devin -o -name .continue -o -name .codex \) -print
```

Interpretation:

- No `.ai/`: use the normal installer.
- Existing `.ai/`: do not run `install.sh`; inspect and plan an upgrade.
- Existing native IDE folders: install first, then run `doctor` and consider `adopt`
  or `reconcile`.

## Step 2: Install The Kit

From this kit repo:

```bash
./install.sh /path/to/target-repo
```

If the installer asks about the pre-commit guard, choose based on the user's preference.
If the user did not specify, installing it is recommended for team repos.

If interactive input is not practical, skip the hook during install and copy it later:

```bash
cp hooks/pre-commit /path/to/target-repo/.git/hooks/pre-commit
chmod +x /path/to/target-repo/.git/hooks/pre-commit
```

## Step 3: Bootstrap Local Projections

In the target repo:

```bash
cd /path/to/target-repo
./sync-agentic.sh bootstrap
```

This generates local IDE projections from `.ai/` and prints a doctor report.

## Step 4: Reconcile Existing Native Config

In the target repo:

```bash
./sync-agentic.sh doctor
```

Then follow the report:

- Native-only single asset:

  ```bash
  ./sync-agentic.sh adopt <ide> <path>
  ```

- Exact duplicate group:

  ```bash
  ./sync-agentic.sh reconcile <kind> <name>
  ```

- All exact duplicate groups:

  ```bash
  ./sync-agentic.sh reconcile --all-exact
  ```

- Same-name different-content conflict: stop and ask which version should become
  canonical, or merge by editing the `.ai/` asset manually.

After adoption or reconciliation:

```bash
./sync-agentic.sh
./sync-agentic.sh --check
```

## Step 5: Seed A Repo With No Agentic Assets

If the repo has no existing rules, commands, skills, or agents, keep the built-in
`agentic-config-maintainer` skill and `/agentic-config` command. Replace or remove only
the starter examples when the user wants repo-specific guidance.

Recommended minimum seed:

1. A repo guidance rule:

   ```text
   .ai/rules/repo-guidance.md
   ```

   Use for durable conventions every agent should know.

2. One useful command:

   ```text
   .ai/commands/summarize-changes.md
   ```

   Use for a repeatable slash/manual workflow.

3. One practical skill:

   ```text
   .ai/skills/<repo-task>/SKILL.md
   ```

   Use for the most common repo-specific workflow.

If the user has not provided repo-specific content, do not invent detailed policy.
Create a small placeholder rule only when they explicitly want a ready-to-edit starter.

## Step 6: Verify

In the target repo:

```bash
./sync-agentic.sh doctor
./sync-agentic.sh --check
```

Expected:

- `--check` reports generated entries are in sync.
- `doctor` may report degraded mappings because not every IDE has identical concepts.
  That is acceptable.
- `doctor` should not report native-only assets, exact duplicate groups, stale output,
  or canonical conflicts unless there is intentional follow-up work.

## Step 7: Clean Optional Local Duplicates

Only when the user asks to clean local generated output:

```bash
./sync-agentic.sh clean
./sync-agentic.sh bootstrap
```

Only when `doctor` reports exact native duplicates already represented in `.ai/` and the
user wants them removed:

```bash
./sync-agentic.sh clean --native-duplicates
./sync-agentic.sh
./sync-agentic.sh --check
```

## Step 8: Commit Source-Only Setup

Recommended source-only commit:

```bash
git add .ai .gitignore AGENTS.md sync-agentic.sh
git status --short
```

Do not force-add generated IDE projections unless the user explicitly wants committed
outputs. In source-only mode, generated folders such as `.cursor/`, `.agents/`,
`.claude/`, `.windsurf/`, `.devin/`, `.continue/`, and `.codex/agents/` should remain
ignored local projections.

## Step 9: Tell The User How To Use It

Give the user this short handoff:

```text
Your repo now has shared `.ai/` agentic config.

After cloning, run:
  ./sync-agentic.sh bootstrap

To add or adopt AI config, ask your model to use the agentic-config skill, or run:
  ./sync-agentic.sh doctor

Commit `.ai/`, `AGENTS.md`, `.ai/.manifest.json`, and `sync-agentic.sh`.
Generated IDE folders are local projections and stay ignored.
```
